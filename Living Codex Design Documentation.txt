Vision and Exp
The Living Codex: Project Overview
Vision: The Intelligence Officer for Your TTRPG Campaign
The Living Codex is a headless, AI-powered Game Master assistant designed to solve the two biggest friction points in tabletop roleplaying games: Lore Amnesia and Note-Taking Fatigue.
It is not a passive wiki or a generic "chat with PDF" tool. It is an active Intelligence Officer that lives inside your Discord server. It listens to your sessions, organizes your narrative assets, and protects your campaign secrets—all without breaking the immersion of the game.
________________


The Core Philosophy
We are building for narrative-first games (Armour Astir, Delta Green, Monster of the Week), not just dungeon crawlers. This shapes every design decision:
1. Assets Over Inventory: We track Factions, Intel, and Clues, not "10x Rations" or "Longswords."
2. Text-First Immersion: No complex graphs or slow web portals. Interaction happens via lightning-fast text commands in Discord.
3. Safety First: The system is Spoiler-Aware and Conflict-Averse. It never reveals a secret to a player, and it never overwrites a Game Master's manual notes.
________________


Key Features
1. The "Silent" Recall (For Players)
* Problem: Players forget NPC names or plot details, forcing them to interrupt the GM to ask, "Who is this guy again?"
* Solution: A player types /codex check "Sky Pirates". The bot returns a private, ephemeral message visible only to them.[a]
* Result: The player gets the context they need ("They are the mercenaries who betrayed you in Session 4") instantly, without stopping the flow of the game.
2. The Spoiler Shield (For GMs)[b]
* Problem: In mystery games like Delta Green, searching a database risks revealing the monster before the players find it.
* Solution: The database strictly separates Public Knowledge (transcripts) from GM Secrets (private notes).
   * Player View: "A dusty storage unit."
   * GM View: "A dusty storage unit. SECRET: Contains a live alien."
3. The Conflict Guard (Data Integrity)
* Problem: Automated tools often destroy manual work. If a GM spends 2 hours writing a villain's backstory, an AI update shouldn't wipe it out.
* Solution: Before syncing any update to Foundry VTT, the system calculates a cryptographic hash of the content. If the GM has manually edited a journal since the last sync, the AI aborts the write and flags it for review.
4. The "Scribe" Pipeline (Automation)
* Problem: Post-game burnout. GMs dread listening to 3 hours of audio to write summaries.
* Solution:
   1. Upload: Drop the session audio into a folder.
   2. Transcribe & Extract: Gemini 1.5 Flash (Audio) and Pro (Logic) extract new NPCs, status changes, and plot points.
   3. Review: The GM receives a "Mission Report" of proposed changes.[c][d]
   4. Sync: With one click, the approved data is pushed to Foundry VTT.
________________


Technical Architecture
The Living Codex is engineered to run on critical infrastructure with zero impact.
* Host: Runs alongside a live Avalanche (AVAX) Validator Node.
* Resource Cap: Hard-limited to 0.5 vCPU and 512MB RAM via Docker.
* Zero Local Inference: No heavy machine learning models run on the server. All processing is offloaded to Google's Gemini API to ensure the validator node never misses a block.
* Stack: Python (Asyncio), SQLite (Local Cache), Discord.py, Foundry MCP Bridge[e].

The Living Codex turns the burden of documentation into a strategic asset, letting the GM focus on the drama and the players focus on the action.

🟢 What Works (Keep This)
1. The Pivot to "Assets" (Story 1.1)
* Why: This is perfect for Armor Astir. "The 4th Fleet" is a narrative tool, not an inventory item. Treating it as an "Asset" with a status (Grounded/Active) aligns perfectly with the game mechanics. Keep this.
2. The "Spoiler Wall" (Story 2.1)
* Why: This is non-negotiable for Delta Green. The distinction between "Public Knowledge" and "GM Secret" is the only feature that makes this safe to use in a mystery game. If this fails, the app kills the fun.
3. The Conflict Guard
* Why: I will repeat this until I die: Never overwrite the GM[i]. This safety feature is the only reason I would install this on my server.[j]


________________


🔴 What Doesn't Work (The "Campaign Killers")
1. The "Visual Clue Web" (Story 1.2)
* The Problem: You are asking a text-based LLM (Gemini) to generate a visual node map in real-time inside Discord.
* The Risk: Hallucination. The AI will draw connections that don't exist ("The Mayor is the Killer") because it misunderstood a joke. A misleading visual is worse than no visual.
* The Fix: Cut the Visuals.
* Return a List of Connections instead.
* Format: "The Mayor hired The Janitor (Confidence: High). The Janitor was seen at The Crime Scene."
* Why: Text is easier to debug and easier for the GM to verbally correct ("Actually, the AI is wrong, the Mayor didn't hire him").
2. The "Multiverse" Tags (Story 2.2)
* The Problem: Asking the GM to tag every single entity with World: Astir Prime vs World: Earth 2026 is tedious data entry. GMs hate data entry. They will forget to tag it, and then the search will fail.
* The Fix: Contextual Search.
* Don't make "World" a database column.
* Just rely on the Session Context. If we are playing Monster of the Week, the AI should prioritize entities from recent MotW sessions.
* Simplification: "Search results from Current Campaign are shown first. Results from Archived Campaigns are shown below."
3. "Source Attribution" Granularity (Story 1.3)
* The Problem: Tracking "Who said what" for every sentence is a massive token cost and processing overhead.[k]
* The Fix: Session-Level Attribution.
* Don't track every sentence. Just track the Session ID.
* Output: "The Idol is safe to touch. (Source: Session 4 Summary)."[l]
* Why: If I need to know who said it, I can look at the Session 4 summary. I don't need the database to store "Cultist Leader, Timestamp 14:02." That's too much.

⚠️ Critical Items for the Final Draft
The Designer must know these three things before writing the code:
1. The "Discord Char-Limit" Trap
Discord messages have a 2,000 character limit.


* Requirement: The bot must summarize. It cannot dump a 5-paragraph bio.
* Design Rule: "The Answer, The Context, The Link."
1. The Answer: "Baron Vrax is your rival."
2. The Context: "He burned the tavern."
3. The Link: "[View Full Journal in Foundry]"[m]
2. The "Proper Noun" Nightmare
The AI will misspell names. "Vrax" becomes "Racks." "Astir" becomes "Aster."


* Requirement: The system needs a "Did You Mean?" fuzzy search.
* Design Rule: If I search /codex search Racks, it should find "Lord Vrax" without me typing it perfectly. If it requires perfect spelling, players will stop using it.
3. The "Permissions" Default
* Requirement: By default, everything is GM Only.
* Why: It is better for the players to see nothing than for them to see a spoiler. The GM should explicitly "Publish" info to the players (or the AI assumes "Public" only if it was spoken by a Player in the transcript).[n]

The system operates in three distinct phases matching the lifecycle of a TTRPG session.

3. UI/UX Specifications (For Engineering)
3.1 The "3-Bullet" Rule (Discord Output)
To prevent "Wall of Text" fatigue, all Discord embeds must strictly follow this format:
         1. Header: Name + Status Emoji (🟢/🔴/💀).
         2. Context: The single most relevant fact or relationship.[y]
         3. Source: "According to [Session X]."
3.2 The "Fuzzy" Threshold
         * Problem: Players will type "Vrax", "Vrecks", or "The Baron."
         * Spec: The search query must use a vector similarity threshold of 0.8[z] (or equivalent fuzzy logic).
         * Fallback: If confidence is low (<0.7), return a "Did you mean?" list:
         * Did you mean: Baron Vrax?
         * Did you mean: The Barracks?
3.3 The "Spoiler Wall" Architecture
The database schema must explicitly separate Public vs Private fields.
         * description_public: Sourced from Session Transcripts (What was said aloud).
         * description_private: Sourced from GM Notes or Foundry GM Layer.
         * Default State: If the source is unknown, default to Private.

4. Edge Cases (Failure States)

         * Scenario: The AI thinks "The King is dead" because a player said "I wish the King was dead."

         * The "Foundry is Asleep" State:
         * Scenario: The Forge server is offline/sleeping.
         * Safety: The /codex commands read from the Local Cache (SQLite). The system does not try to wake the Forge server for simple lookups. Sync operations queue until the server is awake.

The Living Codex is a text-first, AI-powered archivist designed to function as an "Intelligence Officer" for TTRPG campaigns (specifically Armour Astir, Delta Green, and Monster of the Week).
It operates as a Discord Bot backed by a local SQLite database and Google’s Gemini API. It ingests session audio and raw notes, structures them into "Assets" and "Intel," and syncs this data to Foundry VTT.[ac]
Critical Constraint: The system hosts on a live Avalanche (AVAX) Validator Node. It must operate with negligible resource consumption (0.5 vCPU cap) to prevent validation downtime. No local ML inference is permitted.

         3. Conflict Guard: The AI is a scribe, not an author. It must never overwrite a manual edit made by the GM in Foundry VTT.
         4. Text-First: No complex visual node generation. Output must be concise, text-based, and mobile-friendly.

         * Containerization: Docker.
         * Resource Limits:

FR1: The "Intelligence Officer" (Discord Interface)
         * Command: /codex check [query]

         3. Format: Ephemeral Embed (Visible only to requester).

         1. Status: (e.g., "Active," "Grounded," "Deceased").
         2. Context: The most relevant relationship or fact.
         3. Source: Session citation (e.g., "Session 12").
         * Command: /codex add [name] [description]
         * Actor: GM Only.
         * Logic: Creates a row in entities with status='Draft'.
         * Latency: < 1 second.
FR2: The "Scribe" Pipeline (Ingestion)
         * Trigger: File upload to watched directory (/inputs/*.mp3).
         * Step 1: Transcription (API Only)
         * Send audio to Gemini 1.5 Flash.
         * Prompt: "Transcribe. Identify speakers. Output JSON."
         * Constraint: DO NOT use local Whisper.
         * Step 2: Entity Extraction
         * Send Transcript + Current Entity List to Gemini 1.5 Pro.
         * Prompt: "Identify Assets, Factions, and NPCs. Extract status changes. Flag 'Secrets' vs 'Public Events'."
         * Step 3: Diff Generation
         * Output a staged_changes.json file.
         * Constraint: System stops here. It waits for human approval via Discord Button or CLI.
FR3: The "Conflict Guard" (Sync Logic)
         * Trigger: GM approves staged_changes.json.
         * Algorithm:
         1. For each entity E in the update list:
         2. Fetch E_foundry from Foundry VTT API.
         3. Calculate Hash(E_foundry.content).
         4. Compare with DB.last_synced_hash.
         5. IF Match: Safe to update. Push new content to Foundry. Update DB.last_synced_hash.
         6. IF Mismatch: STOP. Log error: "Conflict detected for [Entity Name]." Do not update Foundry.
         7. IF New: Create Journal Entry.

5. Data Model (SQLite Schema)
This schema supports the "Asset" focus and "Spoiler Safety."
SQL
-- Core Entity Table
CREATE TABLE entities (
    id INTEGER PRIMARY KEY,
    uuid TEXT UNIQUE NOT NULL,    -- Internal ID
    name TEXT NOT NULL,
    type TEXT CHECK(type IN ('Asset', 'Faction', 'NPC', 'Location', 'Clue')),
    
    -- Status Tracking (Crucial for Assets)
    status_label TEXT,            -- e.g., "Grounded", "Wounded", "Active"
    status_color TEXT,            -- e.g., "Red", "Green" (For Discord Embeds)
    
    -- The Spoiler Wall
    description_public TEXT,      -- Sourced from transcripts
    description_private TEXT,     -- Sourced from GM Notes
    
    -- Conflict Guard
    foundry_id TEXT,              -- UUID in The Forge
    foundry_hash TEXT,            -- SHA256 of last known Foundry content
    
    last_updated DATETIME DEFAULT CURRENT_TIMESTAMP
);


-- Search Aliases (Fuzzy Matching)
CREATE TABLE aliases (
    alias TEXT,
    entity_id INTEGER,
    FOREIGN KEY(entity_id) REFERENCES entities(id)
);


-- Relationships (The "Web")
CREATE TABLE relationships (
    source_id INTEGER,
    target_id INTEGER,
    rel_type TEXT,                -- 'Ally', 'Rival', 'Owner', 'Location'
    citation TEXT,                -- 'Session 4'
    FOREIGN KEY(source_id) REFERENCES entities(id),
    FOREIGN KEY(target_id) REFERENCES entities(id)
);

1. Executive Summary
The Living Codex is a low-latency, AI-powered archivist designed to run alongside a mission-critical blockchain validator without compromising network performance.
It utilizes a Hybrid Storage Architecture that leverages Google Drive for secure transport (Air Gap), a local 21TB HDD for cold storage, and the system NVMe for high-speed database queries. All heavy compute is offloaded to Google's Gemini API, ensuring the local CPU footprint remains negligible (<0.5 vCPU).
Core Constraint: The Avalanche Validator process (avalanchego) has absolute priority. The Codex operates as a background "scavenger" process with strict resource throttling.
________________


2. High-Level Architecture
The "Air Gapped" Data Flow
The system avoids opening inbound firewall ports by using a Polling (Pull) methodology.
         1. Ingestion (The Dead Drop): GM uploads session audio to Google Drive.
         2. Transport (The Pull): Server polls Drive via rclone, moves audio to the Local 21TB HDD.
         3. Processing (The Brain): Python service streams audio from HDD to Gemini 1.5 Flash API.
         4. Database (The Memory): Structured data (Assets, Intel) is written to SQLite on the NVMe SSD.
         5. Interface (The Voice): Discord Bot queries SQLite to serve Players/GM in real-time.
         6. Sync (The Hand): Validated updates are pushed to Foundry VTT via the MCP Bridge.
________________


3. Infrastructure & Hardware Specs
3.1 Resource Partitioning
To protect the Validator's block verification performance, the Codex is containerized with hard limits.
         * Compute: Docker container capped at 0.5 vCPU.
         * Memory: Docker container capped at 512 MB RAM.
         * Process Priority:
         * CPU: nice -n 19 (Lowest priority).
         * Disk I/O: ionice -c 3 (Idle priority).
         * Network: Egress bandwidth limited to 5 Mbps to preserve gossip bandwidth.

4. Security & Data Integrity
4.1 The "Conflict Guard" (Data Safety)
The system acts as a Scribe, not an Author. It must never overwrite manual work.
         * Mechanism: Optimistic Concurrency Control (OCC).
         * Protocol:
         1. Fetch Journal from Foundry.
         2. Calculate SHA-256 Hash of current content.
         3. Compare with DB.last_synced_hash.
         4. IF Mismatch: Abort Sync. Log "Conflict Detected."
         5. IF Match: Proceed with update.
4.2 The "Spoiler Shield" (Game Integrity)
The database schema explicitly segregates knowledge to protect campaign mysteries (e.g., Delta Green).
         * Field: description_public (Sourced from Session Audio).
         * Field: description_private (Sourced from GM Notes).
         * Access Control: The Discord Bot checks User_ID. Only the GM receives the private field.
4.3 PII & Privacy
         * Redaction: The prompt to Gemini Pro includes a directive to anonymize real-world names in the structured output.
         * Retention: Raw transcripts are processed in memory or ephemeral /tmp storage and deleted immediately after extraction. Only the structured "Intel" is retained in the DB.

5. Application Logic (The "Scribe" Pipeline)
Phase A: Ingest (The Cron Job)
Runs every 10 minutes.
Bash
# Pulls from Drive to HDD, limiting bandwidth
rclone move gdrive:/Codex_Input /mnt/hdd/codex_raw/ --bwlimit 5M


Phase B: Process (The Python Service)
Triggered when new files land in /mnt/hdd.
         1. Read: Stream MP3 from HDD.
         2. Transcribe: Send to Gemini 1.5 Flash.
         3. Extract: Send Transcript to Gemini 1.5 Pro (Context: "Extract Assets & Intel").
         4. Diff: Generate a staged_changes.json file.
         5. Wait: Send "Mission Report" to Discord. Wait for GM Approval.
Phase C: Commit (The Sync)
Triggered by GM Approval.
         1. Write: Update codex.db (NVMe).
         2. Push: Call Foundry API via MCP Bridge (subject to Conflict Guard).
         3. Backup: Snapshot DB and push to Google Drive.
